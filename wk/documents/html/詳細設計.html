Lambda詳細設計書（実務版）

本設計書は、釣行AIアプリにおけるAWS Lambda関数単位の責務・入出力・処理フローを定義する。

1. 設計方針

  • 1 Lambda = 1責務（単一責任原則）
  • API系とバッチ系を分離
  • DynamoDBアクセスは原則Lambda経由
  • リアルタイム処理は禁止（バッチ中心）

【2026-07-24 実装更新・午前】これまで generateSpotScoreBatch はSpotsテーブルの既存スポットをre-scoreするのみで、 新規スポットを増やす手段が無かった（seed_data.pyの手動編集・再実行のみ）。discoverSpotsBatch を追加し、 Google Places API のText Search（実在POIデータ）で新規候補を探索、既存スポットと300m以内なら重複除外した上で Spotsテーブルへ自動追加するようにした。想定される魚種はGemini APIで地名から推測するが、 座標・名称などの実在性に関わる情報はPlaces APIの結果のみを採用し、LLMには生成させない （座標のハルシネーションを避けるため）。
【2026-07-24 実装更新・同日中に再変更】ユーザーからのフィードバックにより、「新スポットを探す」を独立ボタンのまま 残すのではなく、generateSpotScoreBatch が discover_spots.run_discovery() を内部で呼び出す形に統合した。 「AI分析を実行」ボタンを押すたびに (a) 位置指定なしの全国向け探索 → (b) 全スポットのスコア計算、の順に実行される。 これに伴い discoverSpotsBatch の週次EventBridgeスケジュールは冗長になったため廃止し、この関数は 「現在地から新スポットを探す」ボタン（ユーザーの現在地に絞った近傍探索専用、半径15km、スコア計算はしない） 専用として存続させている。

実装上は generate_score.py と discover_spots.py の相互import（循環import）を避けるため、 共有ヘルパー（DynamoDB/SSMアクセス等）を新設の batch_common.py に切り出し、両者がそこから import する構成にした。

4. 個別設計

getRecommendationsHandler

目的：ユーザーにTOP3スポットを返却

入力

{
  "userId": "xxx",
  "lat": 35.6,
  "lng": 139.7
}

処理フロー

1. Recommendations取得
2. 距離補正
3. スコア順ソート
4. TOP3抽出
5. レスポンス返却

出力

{
  "top3": [
    {
      "spotId": "A123",
      "score": 87,
      "fishTypes": ["アジ"],
      "reason": "潮流が安定",
      "distance": 3.2,
      "cost": 500
    }
  ]
}
---

generateSpotScoreBatch

目的：AIによる釣行スコア生成

処理

Spots取得
 + Weatherデータ（Open-Meteo Forecast API）
 + Tideデータ（Open-Meteo Marine API・海面水位）
 → スコア計算
 → Gemini APIで推薦理由(reason)を生成
 → DynamoDB更新

天気・潮汐データソース【2026-07-23 実装】

外部の天気・潮汐APIは Open-Meteo（https://open-meteo.com/）を採用。 APIキー登録不要・無料（非商用）で、複数の気象・海洋機関のデータを統合しており信頼性が高い。
・天気スコア：weather_code（WMO天気コード）・降水量・風速から独自ヒューリスティックで算出
・潮汐スコア：sea_level_height_msl（海面水位）の前後1時間の変化率から算出 　（潮が動いている時間帯ほど魚の活性が上がるという経験則に基づく）
・外部API呼び出しが失敗した場合はシミュレーション値にフォールバックし、バッチ全体は継続する

スコアロジック

score =
  fish_probability * 0.4
+ weather_score * 0.2
+ tide_score * 0.2
- distance_penalty * 0.1
- cost_penalty * 0.1
---

adminRunAiBatch【2026-07-23 追記】

目的：フロントの「AI分析を実行」ボタンから generateSpotScoreBatch を起動する
API Gateway（REST API）は統合タイムアウトが29秒固定であり、Gemini API呼び出しを含む generateSpotScoreBatch（5スポット分）はこれを超えることがある。当初は同期呼び出し （RequestResponse）で実装していたため、Lambda自体は成功しているのに API Gateway が 504 timeoutを返し、ボタンにエラー表示が出る問題があった。

対応として adminRunAiBatch は generateSpotScoreBatch を非同期呼び出し（Event）に変更し、 起動受付（202）を即座に返す方式に変更した。完了確認はフロントエンド側で GET /recommendations を定期的にポーリングして行う。

discover_spots.run_discovery()【2026-07-24 追加】

目的：新規釣りスポット候補をGoogle Places APIで探索し、Spotsテーブルに追加する中核ロジック。 generateSpotScoreBatch のhandler冒頭（位置指定なし）と、discoverSpotsBatch のhandler （「現在地から新スポットを探す」ボタン経由、位置指定あり）の両方から呼ばれる共通関数。

処理

Spots全件取得（重複判定用）
 → Google Places API Text Search
    ・位置指定なし: "釣り 堤防" 等の全国向け複数クエリ
    ・位置指定あり: "釣り" 1クエリ + location/radius(15km)パラメータで近傍に絞り込み
 → 既存スポットと緯度経度がhaversine距離300m以内なら重複として除外
 → Gemini APIで地名・住所から想定される魚種を推測（失敗時は汎用デフォルト）
 → 基準地点（位置指定時はその座標、なければ東京駅）からのdistanceKmを算出
 → Spotsテーブルへput_item
座標・スポット名などの実在性に関わる情報はGoogle Places APIの結果のみを採用し、 LLM（Gemini）には生成させない（存在しない場所をAIが作り出すハルシネーションを避けるため）。 Geminiは魚種の推測という、実在性に影響しない付随情報の生成にのみ用いる。
generate_score.py が discover_spots.py の run_discovery() を直接importして呼び出すため、 逆方向のimport（discover_spots→generate_score）は循環importになり不可能。共有ヘルパー （DynamoDB/SSMアクセス・HTTP GET）は batch_common.py に切り出し、両モジュールがそこから import する。

presignUploadHandler / updateSpotImageHandler / createPostHandler【2026-07-24 追加】

目的：スポット写真・投稿写真のアップロードと、釣果投稿の作成

処理フロー（写真アップロード）

1. フロントが POST /uploads/presign を呼び、S3への署名付きPUT URL（有効期限5分）を取得
2. フロントがそのURLへ画像ファイルを直接PUT（Lambda/API Gatewayを経由しない。ペイロード制限回避のため）
3. アップロード完了後、公開URL（publicUrl）を
   PUT /spots/{spotId}/image （スポット写真）または
   POST /posts のimageUrl（投稿写真）として保存
アップロード先S3バケット（UploadsBucket）はGetObjectのみ公開許可、書き込みは署名付きURL経由のみに限定している。

5. 非機能要件

  • API応答：1〜2秒以内
  • バッチ：1時間単位実行
  • エラーハンドリング：リトライ設計

7. まとめ

Lambdaは「API処理」と「バッチ処理」に完全分離し、 スケーラブルかつ低コストな構成とする。

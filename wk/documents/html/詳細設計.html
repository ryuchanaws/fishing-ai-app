Lambda詳細設計書（実務版）

本設計書は、釣行AIアプリにおけるAWS Lambda関数単位の責務・入出力・処理フローを定義する。

1. 設計方針

  • 1 Lambda = 1責務（単一責任原則）
  • API系とバッチ系を分離
  • DynamoDBアクセスは原則Lambda経由
  • リアルタイム処理は禁止（バッチ中心）
---

4. 個別設計

getRecommendationsHandler

目的：ユーザーにTOP3スポットを返却

入力

{
  "userId": "xxx",
  "lat": 35.6,
  "lng": 139.7
}

処理フロー

1. Recommendations取得
2. 距離補正
3. スコア順ソート
4. TOP3抽出
5. レスポンス返却

出力

{
  "top3": [
    {
      "spotId": "A123",
      "score": 87,
      "fishTypes": ["アジ"],
      "reason": "潮流が安定",
      "distance": 3.2,
      "cost": 500
    }
  ]
}
---

generateSpotScoreBatch

目的：AIによる釣行スコア生成

処理

Spots取得
 + Weatherデータ（Open-Meteo Forecast API）
 + Tideデータ（Open-Meteo Marine API・海面水位）
 → スコア計算
 → Gemini APIで推薦理由(reason)を生成
 → DynamoDB更新

天気・潮汐データソース【2026-07-23 実装】

外部の天気・潮汐APIは Open-Meteo（https://open-meteo.com/）を採用。 APIキー登録不要・無料（非商用）で、複数の気象・海洋機関のデータを統合しており信頼性が高い。
・天気スコア：weather_code（WMO天気コード）・降水量・風速から独自ヒューリスティックで算出
・潮汐スコア：sea_level_height_msl（海面水位）の前後1時間の変化率から算出 　（潮が動いている時間帯ほど魚の活性が上がるという経験則に基づく）
・外部API呼び出しが失敗した場合はシミュレーション値にフォールバックし、バッチ全体は継続する

スコアロジック

score =
  fish_probability * 0.4
+ weather_score * 0.2
+ tide_score * 0.2
- distance_penalty * 0.1
- cost_penalty * 0.1

adminRunAiBatch【2026-07-23 追記】

目的：フロントの「AI分析を実行」ボタンから generateSpotScoreBatch を起動する
API Gateway（REST API）は統合タイムアウトが29秒固定であり、Gemini API呼び出しを含む generateSpotScoreBatch（5スポット分）はこれを超えることがある。当初は同期呼び出し （RequestResponse）で実装していたため、Lambda自体は成功しているのに API Gateway が 504 timeoutを返し、ボタンにエラー表示が出る問題があった。

対応として adminRunAiBatch は generateSpotScoreBatch を非同期呼び出し（Event）に変更し、 起動受付（202）を即座に返す方式に変更した。完了確認はフロントエンド側で GET /recommendations を定期的にポーリングして行う。

5. 非機能要件

  • API応答：1〜2秒以内
  • バッチ：1時間単位実行
  • エラーハンドリング：リトライ設計
---

6. セキュリティ

  • IAM最小権限
  • Cognito認証必須
  • API Gateway認証制御
---

7. まとめ

Lambdaは「API処理」と「バッチ処理」に完全分離し、 スケーラブルかつ低コストな構成とする。
